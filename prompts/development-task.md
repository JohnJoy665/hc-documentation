# Development Task

Прочитай `AGENTS.md`.

Работай в режиме `development mode`.

## Задача пользователя

Ниже пользователь должен вставить конкретную задачу по WebSoft HCM / WebTutor:

```text
<ВСТАВИТЬ ЗАДАЧУ>
```

## Перед реализацией

Перед тем как писать решение или код, изучи локальную базу знаний:

- `index/map.md`
- `index/glossary.md`
- `index/open_questions.md`
- `docs/`
- `rag/`

Не игнорируй локальную документацию в пользу общих знаний.

## Проверка достаточности знаний

Перед написанием кода проверь, хватает ли информации.

Особенно проверь:

- известен ли точный объект или каталог;
- известны ли точные поля;
- известен ли способ поиска записей;
- известен ли способ сохранения изменений;
- известно ли, где должен выполняться код;
- есть ли подтвержденный пример похожего кода;
- поддерживаются ли нужные конструкции внутренним серверным языком WebTutor.

## Важное ограничение

Серверный код WebTutor не является обычным JavaScript.

Запрещено писать решение как Node.js, браузерный JavaScript или современный JavaScript без подтверждения в документации.

Не используй без подтверждения:

- `require`
- `import`
- `export`
- `module.exports`
- `fs`
- `path`
- `process`
- `fetch`
- `Promise`
- `async/await`
- npm-пакеты
- DOM API
- browser API
- Node.js API

## Если знаний не хватает

Не пиши финальный код.

Ответь так:

```markdown
# Не хватает данных для безопасной реализации

## Что уже известно из документации

- ...

## Чего не хватает

- ...

## Какие материалы нужно добавить в INBOX

- ...

## Следующий шаг

Добавьте материалы в `INBOX/`, затем запустите `documentation mode` через `prompts/process-inbox.md`.

После обновления документации можно продолжить задачу.
```

## Если знаний хватает

Сначала дай план, затем код.

Формат:

```markdown
# План реализации

1. ...
2. ...
3. ...

# Важные предположения

- ...

# Код

...

# Что проверить перед запуском

- ...
```

Для массовых изменений по умолчанию предусматривай:

- dry-run режим;
- логирование;
- фильтр отбора;
- ограничение количества записей для теста;
- осторожное сохранение изменений;
- список проверок перед запуском.
