# WebTutor / WebSoft HCM: Агенты сервера и server-side-java-script

## 1. Краткое назначение

**Агенты сервера** — механизм WebTutor / WebSoft HCM для автоматического запуска процедур по расписанию.

Агенты применяются для выполнения серверных операций, в том числе длительных процедур, в фоновом режиме и обычно в периоды наименьшей нагрузки на сервер.

## 2. Где находятся агенты сервера в системе

Агенты сервера представлены в системе в виде карточек.

Список агентов расположен в административном интерфейсе:

```text
Дизайнер → Бизнес-логика → Агенты сервера
```

При необходимости агенты могут быть разбиты на категории.

В библиотеке открытых компонентов WebSoft можно найти готовые агенты. Их можно использовать как готовые решения или как примеры для разработки собственных агентов.

## 3. Важное отличие от HTML-шаблонов

В HTML-шаблонах WebTutor серверный код обычно пишется внутри конструкций:

```html
<% ... %>
```

В файлах агентов сервера код пишется **без** этих конструкций.

То есть в агенте не нужно оборачивать серверный код в `<% ... %>`.

## 4. Расширение файлов агентов

Файлы серверных агентов обычно имеют расширение:

```text
.js
```

При этом важно: несмотря на расширение `.js`, код агента не является обычным браузерным JavaScript или Node.js JavaScript.

## 5. Язык серверных агентов

Код серверных агентов пишется на внутреннем языке WebTutor, который в документации и практике часто называют:

```text
server-side-java-script
```

Критически важно для AI/Codex:

- это **не JavaScript** в смысле браузерного JS;
- это **не Node.js**;
- это внутренний серверный язык WebTutor / WebSoft HCM;
- синтаксис похож на JavaScript, но доступные объекты, функции и поведение определяются платформой WebTutor;
- нельзя автоматически применять практики обычного JavaScript без проверки по документации WebTutor;
- при отсутствии точного описания функции, объекта или метода в локальной документации нужно запросить дополнительную документацию, а не додумывать по аналогии с JS.

## 6. Пример кода агента сервера

Ниже пример серверного агента. Код расположен непосредственно в файле агента и не оборачивается в `<% ... %>`.

```js
var testId = 7402958296595208218;
oRequest = ArrayOptFirstElem(XQuery('for $elem in requests where $elem/object_id = '+OptInt(testId)+' and $elem/status_id = \'pay\' return $elem'));
if (oRequest != undefined) {
  iReqId = OptInt(oRequest.id);
  oRequestDoc = OpenDoc( UrlFromDocID(iReqId));
  oRequestDocTE = oRequestDoc.TopElem;
  oRequestDocTE.status_id = 'close';
  curState = String(oRequestDocTE.workflow_state);
  oBossArr =  ArrayOptFindByKey(oRequestDocTE.workflow_log_entrys, curState, 'finish_state');
  if (oBossArr != undefined) {
    alert('id boss = ' + oBossArr.person_id);
    alert('id test = ' + testId);
    test = tools.create_notification('notifAfterSuccTest', OptInt(oBossArr.person_id), '', OptInt(testId));
    alert('Отправка уведомления после тестирования = ' + test);
  }
  oRequestDoc.Save();
}

alert('ebanniy pizdec');
```

## 7. Что делает пример

Пример демонстрирует типичный сценарий серверного агента:

1. Задан идентификатор объекта `testId`.
2. Через `XQuery` выполняется поиск заявки в каталоге `requests`.
3. Ищется заявка, у которой:
   - `object_id` равен `testId`;
   - `status_id` равен `pay`.
4. Если заявка найдена:
   - берётся ID заявки;
   - открывается документ через `OpenDoc(UrlFromDocID(iReqId))`;
   - получается `TopElem` документа;
   - статус заявки меняется на `close`;
   - определяется текущее состояние workflow;
   - в журнале workflow ищется запись с нужным `finish_state`;
   - если запись найдена, берётся `person_id` руководителя/ответственного;
   - создаётся уведомление через `tools.create_notification(...)`;
   - документ сохраняется через `oRequestDoc.Save()`.

## 8. Использованные платформенные функции и объекты

В примере используются платформенные функции и объекты WebTutor / WebSoft HCM:

- `XQuery(...)` — выполнение XQuery-запроса по данным WebTutor;
- `ArrayOptFirstElem(...)` — получение первого элемента массива или `undefined`;
- `OptInt(...)` — безопасное приведение значения к целому числу;
- `OpenDoc(...)` — открытие документа по URL документа;
- `UrlFromDocID(...)` — получение URL документа по ID;
- `TopElem` — верхний элемент открытого документа;
- `ArrayOptFindByKey(...)` — поиск элемента массива по значению поля;
- `tools.create_notification(...)` — создание уведомления;
- `Save()` — сохранение изменённого документа;
- `alert(...)` — штатный способ вывести диагностические данные в консоль / вывод выполнения серверного агента WebTutor. В контексте агента это не браузерный `alert`, а практический механизм серверной отладки.

Если точное поведение функции не описано в локальной документации, его нужно уточнять по документации WebTutor, а не переносить поведение из обычного JavaScript.

## 9. Практические правила для разработки агентов

При создании или изменении серверных агентов WebTutor нужно учитывать:

- код агента пишется без `<% ... %>`;
- файл может иметь расширение `.js`, но это не означает использование обычного JavaScript;
- код выполняется на стороне сервера WebTutor;
- доступны платформенные функции WebTutor, например `XQuery`, `OpenDoc`, `UrlFromDocID`, `OptInt`, `tools.*`;
- изменение документов нужно завершать сохранением через `Save()`;
- `alert(...)` можно использовать для диагностики выполнения агента: вывода ID, промежуточных значений, результата вызова функции и других проверочных данных;
- важно не путать `alert(...)` в серверном агенте WebTutor с браузерным `window.alert`: в агенте это способ вывести данные в консоль / лог выполнения агента;
- при работе с объектами, документами, каталогами и workflow необходимо сверяться с документацией по конкретным типам объектов и полям.

## 10. AI/Codex instructions

При обработке этого материала AI-агент должен соблюдать следующие правила:

1. Не считать код серверного агента обычным JavaScript.
2. Не предлагать Node.js API, browser API, `fetch`, `Promise`, `async/await`, npm-пакеты и другие внешние JS-механизмы, если они прямо не подтверждены документацией WebTutor.
3. Не оборачивать код агента в `<% ... %>`.
4. При генерации кода агента использовать стиль server-side-java-script WebTutor.
5. При неизвестных функциях или объектах запрашивать дополнительную документацию.
6. Не переписывать платформенные функции WebTutor на обычный JS без прямого указания.
7. Не удалять `alert(...)` автоматически как "мусорную отладку": в WT-агентах это реальный способ вывести данные при выполнении агента. Можно помечать такие строки как диагностические, но не менять их смысл без запроса.
8. Сохранять различие между:
   - HTML-шаблонами WebTutor, где серверный код пишется внутри `<% ... %>`;
   - серверными агентами, где код пишется напрямую в файле агента.

## 11. Ключевые выводы

- Агент сервера — серверная процедура WebTutor, запускаемая автоматически по расписанию.
- Агент удобен для фоновых и длительных операций.
- Код агента пишется без `<% ... %>`.
- Файлы агентов имеют расширение `.js`, но язык — внутренний `server-side-java-script`, а не обычный JavaScript.
- Для корректной разработки нужно опираться на документацию WebTutor и платформенные функции.
