## Назначение

XQuery в SP-XML используется для выборки данных из каталогов базы данных WebTutor/WebSoft HCM.

Важно: в SP-XML используется не полный стандартный XQuery, а ограниченное подмножество синтаксиса, дополненное платформенными функциями.

Основной рабочий сценарий — получить записи из одного или нескольких каталогов, отфильтровать их через `where`, отсортировать через `order by` и вернуть запись или отдельные поля через `return`.

## Общая форма запроса

```xquery
for <декларация переменных>
(where <условие>)
(order by <имя поля> (descending | ascending))
return <возвращаемое значение>
```

Круглые скобки в описании синтаксиса означают необязательные части запроса.

Типовой запрос:

```xquery
for $elem in events
where $elem/status_id = 'plan'
order by $elem/start_date descending
return $elem
```

## Грамматика запроса

```text
<декларация переменных> = <декларация переменной> {, <декларация переменных>}

<декларация переменной> = <имя переменной> in <имя каталога>

<имя переменной> = начинается с $, затем любая последовательность латинских букв

<условие> = <выражение 1> <логический оператор> <выражение 2> <логический оператор> <выражение 3> ...

<логический оператор> = and | or

<выражение> = <величина> <оператор сравнения> <величина>

<оператор сравнения> = > | >= | < | <= | = | != | MatchSome(<выражение> {, <выражение>}) | contains(<выражение>, <выражение>) | doc-contains(<выражение>)

<величина> = <имя поля> | <константа> | date(<величина>) | null() | true() | false()

<имя поля> = <имя переменной>/<имя поля каталога>

<возвращаемое значение> = <имя переменной> | <имя переменной>/<имя поля каталога> | <имя переменной> {, <возвращаемое значение>}
```

## Декларация переменных

Декларация переменной связывает переменную с каталогом базы данных.

```xquery
for $x in groups return $x
```

Здесь:

| Элемент | Значение |
|---|---|
| `$x` | переменная запроса |
| `groups` | каталог базы данных |
| `$x/...` | обращение к полям записи из каталога `groups` |

Можно объявлять несколько переменных, если запрос связывает записи из нескольких каталогов.

```xquery
for $e in education_methods, $x in events
where $x/education_method_id = $e/id and $x/status_id = 'plan'
return $e
```

Такой запрос работает как соединение каталогов через условие в `where`.

## Условия `where`

Блок `where` содержит логическое условие.

Поддерживаются логические операторы:

| Оператор | Назначение |
|---|---|
| `and` | логическое И |
| `or` | логическое ИЛИ |

Поддерживаются операторы сравнения:

| Оператор | Назначение |
|---|---|
| `=` | равно |
| `!=` | не равно |
| `>` | больше |
| `>=` | больше или равно |
| `<` | меньше |
| `<=` | меньше или равно |

Пример:

```xquery
for $c in events
where $c/status_id = 'plan' and $c/type_id = 'webinar'
return $c
```

## Сортировка `order by`

Блок `order by` задает поле сортировки.

```xquery
for $c in events
where $c/status_id = 'plan'
order by $c/start_date descending
return $c
```

Поддерживаемые направления сортировки:

| Направление | Значение |
|---|---|
| `ascending` | по возрастанию |
| `descending` | по убыванию |

Если направление не указано, следует исходить из поведения платформы по умолчанию. Для AI-документации безопаснее явно указывать направление сортировки, когда оно важно для результата.

## Возвращаемое значение `return`

`return` может возвращать:

- всю запись каталога;
- отдельное поле записи;
- несколько значений.

Вернуть всю запись:

```xquery
for $elem in events return $elem
```

Вернуть поле записи:

```xquery
for $elem in events return $elem/name
```

Вернуть набор полей через платформенную конструкцию `Fields()`:

```xquery
for $elem in events return $elem/Fields('name','type_id')
```

## Функции внутри XQuery

### `contains()`

Ищет подстроку в строке.

```xquery
contains(arg1, arg2)
```

Аргументы:

| Аргумент | Описание |
|---|---|
| `arg1` | величина, внутри которой выполняется поиск |
| `arg2` | величина, содержащая искомую подстроку |

Пример:

```xquery
for $emp in collaborators
where contains($emp/position_name, 'инженер')
return $emp
```

Особенность: по исходной документации `contains()` игнорирует регистр.

Предупреждение: функцию следует использовать осторожно, потому что ее выполнение может не оптимизироваться индексами встроенной СУБД.

### `doc-contains()`

Выполняет полнотекстовый поиск в документе.

```xquery
doc-contains(arg1, arg2, arg3)
```

Аргументы:

| Аргумент | Описание |
|---|---|
| `arg1` | имя поля, в котором содержится ID документа, где будет выполняться полнотекстовый поиск |
| `arg2` | имя базы для поиска; часто используется пустая строка `''` или `DefaultDb` |
| `arg3` | величина, содержащая искомую строку |

Пример:

```xquery
for $elem in candidates
where doc-contains($elem/id, 'data', 'xxx')
return $elem
```

Также встречается форма с указанием каталога:

```xquery
for $elem in candidates
where doc-contains($elem/id, 'data', 'xxx', 'candidate')
return $elem
```

### `MatchSome()`

Проверяет пересечение значения поля или последовательности значений с заданным множеством.

```xquery
MatchSome(arg1, arg2)
```

Аргументы:

| Аргумент | Описание |
|---|---|
| `arg1` | поле или последовательность, где выполняется поиск |
| `arg2` | массив/последовательность значений для поиска вида `(<величина>{,<величина>})` |

Пример:

```xquery
for $elem in events
where MatchSome($elem/type_id, ('one_time', 'real_time'))
return $elem
```

Еще пример:

```xquery
for $elem in candidates
where MatchSome($elem/profession_id, (5618059798961344771, 2918050598961344456))
return $elem
```

### `some ... satisfies`

Конструкция используется для поиска элементов из одного каталога, для которых элементы из другого каталога соответствуют указанному критерию.

Общая форма:

```xquery
for $elem1 in <каталог1>
where some $elem2 in <каталог2> satisfies (<критерий>)
return $elem1
```

Эта конструкция полезна, когда нужно проверить существование связанной записи в другом каталоге.

### `null()`

Возвращает значение `null` и используется для проверки пустого поля.

Пример условного шаблона:

```xquery
for $elem in collaborators
where $elem/some_field = null()
return $elem
```

В XQuery SP-XML `null()` используется как функция-литерал, потому что отдельной константы `null` в синтаксисе запроса нет.

### `true()`

Возвращает булевское значение `true`.

Пример:

```xquery
for $x in groups
where $x/is_dynamic = true()
return $x
```

### `false()`

Возвращает булевское значение `false`.

Пример:

```xquery
for $er in event_results
where $er/is_assist = false()
return $er
```

### `date()`

Приводит величину к дате для сравнения с полем типа дата.

```xquery
date(arg1)
```

Аргументы:

| Аргумент | Описание |
|---|---|
| `arg1` | величина, содержащая строку, приводимую к дате |

Пример:

```xquery
for $elem in events
where $elem/start_date > date('01.06.2019 15:30:30')
return $elem
```

В других разделах документации также встречается формат ISO:

```xquery
for $elem in vacancies
where $elem/start_date >= date('2011-06-27')
  and $elem/start_date <= date('2011-06-28T23:59:59')
return $elem
```

Практическое правило: для новых примеров предпочтительно использовать ISO-подобный формат `YYYY-MM-DD` или `YYYY-MM-DDTHH:MM:SS`, если нет требования совместимости с существующим кодом.

## Поиск по кастомным полям через `doc-contains()`

Через `doc-contains()` можно выполнять поиск по кастомным полям.

Общий принцип:

```xquery
doc-contains($elem/id, DefaultDb, '[<имя кастомного поля>=<значение критерия поиска>~<тип кастомного поля>]')
```

Примеры типов кастомного поля:

| Тип | Описание |
|---|---|
| `string` | строка |
| `date` | дата |
| `integer` | целое число |
| `bool` | булево значение |

Пример поиска по кастомному полю:

```xquery
doc-contains($elem/id, DefaultDb, '[custom_field_name=value~string]')
```

Синтаксис теоретически позволяет описывать не только поиск по тексту, но и условия:

```xquery
doc-contains($elem/id, DefaultDb, '[custom_field_name>2023-12-12~date]')
```

Важное ограничение: такие условия работают только на реляционных СУБД, например MS SQL или PostgreSQL. На базе типа XML такой вариант работать не будет.

## Примеры запросов

### Динамические группы

```xquery
for $x in groups
where $x/is_dynamic = true()
return $x
```

Выбирает динамические группы.

### Поиск сотрудников по названию должности

```xquery
for $emp in collaborators
where contains($emp/position_name, 'инженер')
return $emp
```

Выбирает сотрудников, у которых поле `position_name` содержит подстроку `инженер`.

### Плановые вебинары с сортировкой по дате

```xquery
for $c in events
where $c/status_id = 'plan' and $c/type_id = 'webinar'
order by $c/start_date descending
return $c
```

Выбирает плановые мероприятия типа `webinar` и сортирует их по дате начала в обратном порядке.

### Выбор мероприятий по нескольким типам

```xquery
for $elem in events
where MatchSome($elem/type_id, ('one_time', 'real_time'))
return $elem
```

Выбирает мероприятия, у которых `type_id` входит в заданный набор.

### Сравнение даты

```xquery
for $elem in events
where $elem/start_date > date('01.06.2019 15:30:30')
return $elem
```

Выбирает мероприятия, начавшиеся после указанной даты.

### Join каталогов `education_methods` и `events`

```xquery
for $e in education_methods, $x in events
where $x/education_method_id = $e/id and $x/status_id = 'plan'
return $e
```

Выбирает методы обучения, связанные с плановыми мероприятиями.

### Поиск сотрудниц по имени и сортировка по дате рождения

```xquery
for $elem in collaborators
where $elem/sex = 'w' and contains($elem/fullname, 'Анна')
order by $elem/birth_date descending
return $elem
```

Выбирает сотрудников женского пола, у которых ФИО содержит `Анна`, и сортирует по дате рождения.

### Join трех каталогов

```xquery
for $er in event_results, $cl in collaborators, $ev in events
where $er/person_id = $cl/id
  and $er/event_id = $ev/id
  and $ev/status_id = 'close'
  and $er/is_assist = false()
return $cl
```

Выбирает сотрудников, связанных с результатами завершенных мероприятий, где признак участия `is_assist` равен `false`.

### Возврат ограниченного набора полей

```xquery
for $elem in events
return $elem/Fields('name','type_id')
```

Возвращает не всю запись, а выбранные поля.

## Практические правила для разработки

1. Не считать XQuery в SP-XML полноценным стандартным XQuery.
2. Для новых запросов использовать простой шаблон `for ... where ... order by ... return`.
3. Для булевых значений использовать `true()` и `false()`, а не строковые значения `'true'` / `'false'`.
4. Для пустых значений использовать `null()`.
5. Для сравнения дат использовать `date()`.
6. Для выбора по набору значений использовать `MatchSome()`.
7. Для поиска подстроки использовать `contains()`, но помнить о возможных проблемах производительности.
8. Для полнотекстового поиска и кастомных полей использовать `doc-contains()`.
9. Для соединения каталогов объявлять несколько переменных в `for` и связывать их через `where`.
10. Для проверки существования связанных записей можно использовать `some ... satisfies`.
11. При генерации кода для WebTutor проверять имена каталогов и полей по конкретной конфигурации/документации проекта.

## Краткая сводка для Codex

- Основной синтаксис XQuery в SP-XML:

```xquery
for $elem in catalog
where <condition>
order by $elem/field descending
return $elem
```

- Переменная начинается с `$`, например `$elem`, `$x`, `$cl`.
- Каталог указывается после `in`, например `collaborators`, `events`, `groups`.
- Поля записи читаются через `$elem/field_name`.
- Несколько каталогов можно связывать через несколько переменных:

```xquery
for $a in catalog_a, $b in catalog_b
where $a/foreign_id = $b/id
return $a
```

- Для булевых значений использовать `true()` и `false()`.
- Для пустого значения использовать `null()`.
- Для дат использовать `date()`.
- Для поиска по нескольким значениям использовать `MatchSome()`.
- Для поиска подстроки использовать `contains()`, но осторожно из-за производительности.
- Для полнотекстового поиска и поиска по кастомным полям использовать `doc-contains()`.
- Условия `doc-contains()` по кастомным полям с операторами сравнения зависят от типа СУБД: на MS SQL/PostgreSQL могут работать, на XML-базе — нет.
