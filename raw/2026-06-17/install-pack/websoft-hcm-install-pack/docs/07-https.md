# Настройка HTTPS для WebSoft HCM

Для защиты данных и соблюдения требований информационной безопасности портал WebSoft HCM должен обслуживаться по протоколу HTTPS. В базе знаний приводятся разные сценарии настройки в зависимости от способа установки: запуск как сервис Windows, развёртывание в Docker и работа клиентского приложения **Workspace Desktop** с самоподписанными сертификатами. В этом разделе собраны основные рекомендации для каждого варианта.

## HTTPS при запуске как сервис Windows

При запуске WebSoft HCM в режиме службы (Kestrel) требуется прописать параметры HTTPS в конфигурационном файле `xhttp_config.json` и настроить права на сертификат:

1. **Редактирование `xhttp_config.json`.** В разделе `HttpsInlineCertStore` нужно описать конфигурацию HTTPS:

   ```json
   "HttpsInlineCertStore": {
     "Uri": "https://*:443",
     "Address": "0.0.0.0",
     "Certificate": {
       "Subject": "CN=portal.example.com",
       "Store": "My",
       "Location": "LocalMachine",
       "AllowInvalid": "false"
     },
     "Protocols": "Http1AndHttp2"
   }
   ```

   * `Uri` указывает порт (обычно 443); `Address` — IP‑адрес или `0.0.0.0` для прослушивания всех интерфейсов; `Subject` должен соответствовать полю **Issued To** в сертификате; `Store` и `Location` — хранилище, где находится сертификат (как правило, **Personal** / `My` и `LocalMachine`)【949034845440939†screenshot】.
   * Параметр `Protocols` задаёт поддерживаемые версии HTTP: `Http1`, `Http2` или `Http1AndHttp2`【845978304387043†screenshot】.

2. **Назначение сертификата.** В консоли управления сертификатами (mmc.exe → Certificates → Local Computer) найдите нужный сертификат. В его свойствах предоставьте учётной записи службы WebSoft HCM права на чтение приватного ключа и аналогичные права на каталог `C:\ProgramData\Microsoft\Crypto\RSA\MachineKeys`【272105003921069†screenshot】.

3. **Создание узла в администраторе.** В интерфейсе WebSoft Admin перейдите в раздел «Системная Администрация → Узлы» и создайте новый узел с портом 443 и доменным именем портала. Это позволит проксировать запросы через SSL【491657836806193†screenshot】.

4. **Перезапуск сервиса.** После изменений перезапустите службу WebSoft HCM. Убедитесь, что в файле `SpXml.ini` закомментирована строка `;LDS-HTTPS: 1` (или удалена), чтобы включить режим SSL【63072221964003†screenshot】.

## HTTPS в Docker

Для контейнерного развёртывания сервер работает внутри Kestrel; настройка HTTPS выполняется через отдельный конфигурационный файл и параметры запуска контейнера:

1. **Подготовьте конфигурацию.** Скачайте шаблон `xhttp_config.json` из документации (файл `xhttp_config.zip`) и поместите его вместе с сертификатом `.pfx` в директорию `../docker/WebSoftServerForDocker/`.
2. **Отредактируйте файл.** Укажите путь к сертификату и пароль:

   ```json
   "https": {
     "url": "https://*:443",
     "address": "0.0.0.0",
     "certificate": {
       "path": "/WebSoftServer/cert.pfx",
       "password": "пароль"
     },
     "allowInvalid": "true"
   }
   ```【492120819277264†screenshot】

3. **Измените параметры запуска контейнера.** В файле запуска (скрипт `run.sh` или `docker run` командная строка) нужно добавить проброс порта и монтирование конфигурации и сертификата:

   ```bash
   docker run -d --name hcm --restart=always \
     -p 80:80 \
     -p 443:443/tcp \
     -v ~/WebSoftHCM/docker/WebSoftServerForDocker/xhttp_config.json:/WebSoftServer/xhttp_config.json \
     -v ~/WebSoftHCM/docker/WebSoftServerForDocker/cert.pfx:/WebSoftServer/cert.pfx \
     websoft/webtutor-hcm:latest
   ```

   В руководстве также приводится пример модификации файла `docker-compose.yml` с аналогичными параметрами и монтированием сертификата【231624825877915†screenshot】.

4. **Перезапустите контейнер.** После обновления конфигурации найдите идентификатор контейнера (`docker ps`) и перезапустите его (`docker restart <id>`), чтобы настройки вступили в силу.

## Самоподписанный сертификат для Workspace Desktop

Клиентское приложение Workspace Desktop может потребовать доверия к самоподписанному сертификату. В соответствующей статье (не полностью рассмотрена в данном пакете) приведены инструкции по добавлению сертификата в хранилище доверенных корневых центров сертификации Windows и настройке приложения для работы с нестандартным портом. При использовании самоподписанных сертификатов убедитесь, что на сервере включён HTTPS и настроены правильные Subject Name/Alternative Name.

## Рекомендации

* Используйте сертификаты от доверенного удостоверяющего центра. Самоподписанные сертификаты подходят только для тестовых стендов и внутренних систем.
* Включите **HSTS** и другие защитные заголовки на уровне веб‑сервера (см. раздел 12 «Безопасность»). Это усилит защиту от атак типа man‑in‑the‑middle.
* При смене сертификата не забудьте обновить параметры `Subject`/`path` и перезапустить сервис или контейнер.
