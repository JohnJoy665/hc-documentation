# SP-XML: структура приложения, модули и библиотеки

## Кратко

Источник `spxml_application_structure.md` описывает структуру SP-XML приложения: кэш документов, конфигурационные файлы, файл описания приложения, этапы загрузки модулей, модули и библиотеки V1/V2.

## Кэш документов

XML-документы с общими данными обычно загружаются один раз и хранятся в общем списке - кэше документов.

В кэш попадают:

- общие документы из секции `shared_doc` / `shared_docs` модуля;
- v1-библиотеки;
- документы, описывающие разделы UI;
- документы, явно загруженные через `FetchDoc()`.

## Файл описания структуры приложения

Основной модуль приложения должен содержать файл:

```text
xxx_application.xml
```

где `xxx` - имя основного модуля.

Файл содержит:

- все модули приложения (`modules`);
- используемые базы данных (`databases`);
- базу данных по умолчанию (`default_db`);
- стартовый URL (`start_url`).

Стартовый URL указывает на файл, который загружается при открытии основного окна программы. В источнике указано, что обычно это `xxx_view_main.xml`, связанный с `xxx_view_main.xms`.

## Этапы загрузки приложения

Источник описывает несколько проходов загрузки:

1. Загрузка спецификаций модулей: `xxx_module.xml`, языковые константы, `OnModuleLoadUiText()`.
2. Загрузка модулей: регистрация шаблонных форм, `OnModuleLoad()`.
3. Подготовка модулей: регистрация `shared_docs`, `OnModulePrepare()`.
4. Инициализация модулей: регистрация типов объектов, `OnModuleInit()`.
5. Запуск модулей: `OnModuleStart()`.

## Модули

Модуль - папка с программными файлами и описаниями форм. Имя модуля совпадает с именем папки.

Для файлов модулей используется схема:

```text
x-app://rcr/rcr_candidate.xmd
```

Источник указывает, что старые приложения используют `x-local://`, но в новой объектной модели это не рекомендуется.

Каждый модуль должен содержать:

```text
xxx_module.xml
```

Обычно рядом есть:

```text
xxx_module.js
```

с методами инициализации модуля.

## Библиотеки V1 и V2

Библиотеки V1:

- являются XML-документами, часто с `INLINE-FORM`;
- функции представлены методами с параметрами;
- глобальные переменные представлены полями с данными;
- код может быть inline в `EXPR` или в отдельном файле через `CODE-LIB` и `CODE-LIB-URL`;
- могут загружаться через `shared_docs`, `SHARE`, `RegisterSharedDoc()`, `OpenCodeLib()` или `OpenCodeLibDoc()`.

Библиотеки V2:

- являются отдельными объектами-библиотеками;
- могут состоять из `.xmi`, `.js` или `.bs` файлов;
- файлы задают namespace;
- файлы с одним namespace объединяются в одну библиотеку при регистрации;
- поддерживают классы, типизированные параметры и необязательные аргументы через `?`.

## Важные отличия V1 и V2

- В V2 функции могут располагаться в разных файлах.
- V2 поддерживает классы.
- V2 поддерживает типизацию параметров и глобальных переменных.
- В V1 лишние аргументы игнорируются, а недостающие передаются как `undefined`.
- В V2 недостающие аргументы допустимы только если явно помечены как необязательные; лишние аргументы вызывают ошибку.
