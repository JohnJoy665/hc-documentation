# AGENTS.md

Ты работаешь с репозиторием знаний по WebSoft HCM / WebTutor.

Репозиторий предназначен для накопления AI-friendly документации, RAG-ready материалов и практических знаний по разработке, настройке и исследованию WebSoft HCM / WebTutor.

Основной язык документации — русский.

---

# 1. Главная идея репозитория

Пользователь не обязан вручную раскладывать материалы по структуре.

Все новые исходные материалы пользователь складывает в папку `INBOX/`.

Твоя задача — самостоятельно анализировать новые материалы, определять их тему, пополнять документацию, обновлять индекс и создавать атомарные RAG-ready заметки.

Репозиторий должен постепенно превращаться в рабочую базу знаний, которую можно использовать для решения практических задач разработки в WebSoft HCM / WebTutor.

Главный цикл работы:

```text
новые материалы
→ INBOX/
→ documentation mode
→ raw/docs/rag/index обновлены
→ development mode использует обновленную базу знаний
```

---

# 2. Базовая структура репозитория

Используй следующую структуру:

```text
INBOX/
raw/
docs/
rag/
index/
scripts/
prompts/
```

Назначение папок:

- `INBOX/` — новые необработанные материалы, которые добавляет пользователь.
- `raw/` — сохраненные исходники, разложенные по датам и темам.
- `docs/` — связная документация для человека и ИИ.
- `rag/` — короткие атомарные RAG-ready материалы.
- `index/` — карта знаний, источники, терминология, открытые вопросы.
- `scripts/` — вспомогательные скрипты, если они понадобятся.
- `prompts/` — готовые промпты для типовых режимов работы.

Если нужной папки нет — создай ее.

---

# 3. Режимы работы

В репозитории есть два основных режима работы:

1. `documentation mode` — обработка новых материалов из `INBOX/`.
2. `development mode` — решение конкретных задач разработки по WebSoft HCM / WebTutor на основе уже собранной документации.

Если пользователь явно просит обработать новые материалы, обновить документацию, разобрать `INBOX`, создать RAG или структуру — работай в `documentation mode`.

Если пользователь просит написать код, агента, серверный HTML-хендлер, шаблон, обработчик, интеграцию, скрипт или решить прикладную задачу WebTutor — работай в `development mode`.

---

# 4. Важное ограничение: серверный язык WebTutor не является JavaScript

Код, выполняемый на стороне сервера WebSoft HCM / WebTutor, не должен автоматически считаться JavaScript.

Несмотря на внешнюю схожесть синтаксиса с JavaScript, серверный язык WebTutor является внутренним языком платформы со своими объектами, API, функциями, особенностями выполнения и ограничениями.

## 4.1. Жесткое правило

Запрещено писать серверный код для WebTutor как обычный JavaScript, Node.js или браузерный JavaScript.

Нельзя использовать конструкции, библиотеки и API JavaScript/Node.js без подтверждения в локальной документации проекта.

Запрещено автоматически применять:

- `require`
- `import`
- `export`
- `module.exports`
- `fs`
- `path`
- `process`
- `fetch`
- `Promise`
- `async/await`
- npm-пакеты
- DOM API
- browser API
- Node.js API
- современные JavaScript-паттерны, если они не подтверждены документацией WebTutor

## 4.2. Правило при недостатке знаний

Если для конкретной задачи нет подтвержденной инструкции в `docs/`, `rag/`, `index/` или исходных материалах, нельзя додумывать реализацию по аналогии с JavaScript.

В этом случае нужно остановиться и запросить дополнительные материалы.

Формат ответа:

```markdown
# Не хватает данных по серверному языку WebTutor

Для этой задачи недостаточно подтвержденной информации в локальной документации.

## Чего не хватает

- ...

## Какие материалы нужно добавить в INBOX

- пример похожего агента;
- карточку объекта или каталога;
- документацию по нужному методу/функции;
- пример серверного HTML-хендлера;
- пример сохранения объекта;
- пример XQuery-запроса;
- описание нужных полей.

## Следующий шаг

Добавьте материалы в `INBOX/`, затем запустите `documentation mode` для обновления базы знаний.
```

## 4.3. Как писать код для WebTutor

Код для WebTutor можно писать только на основании:

- уже подтвержденных примеров из локальной документации;
- материалов из `docs/`;
- материалов из `rag/`;
- исходников из `raw/`;
- явно добавленных пользователем примеров;
- проверенных фрагментов из предыдущих экспериментов.

Если пример кода выглядит как JavaScript, все равно нужно считать его кодом внутреннего серверного языка WebTutor, а не обычным JavaScript.

## 4.4. Формулировки в документации

Не писать:

- “JavaScript-код на сервере”
- “Node.js-код”
- “обычный JS”
- “серверный JavaScript”, если это не подтверждено источником

Предпочтительные формулировки:

- “серверный код WebTutor”
- “внутренний серверный язык WebTutor”
- “код в серверном HTML-шаблоне”
- “код агента WebTutor”
- “WebTutor-подобный синтаксис”
- “синтаксис, похожий на JavaScript, но выполняемый внутренним механизмом WebTutor”

# 5. Documentation mode

## 5.1. Назначение

В этом режиме ты обрабатываешь новые материалы из `INBOX/` и обновляешь базу знаний.

В `INBOX/` могут лежать:

- `.md`
- `.txt`
- `.pdf`
- `.html`
- `.htm`
- `.json`
- `.xml`
- `.tgz`
- `.zip`
- изображения и скриншоты
- папки с распакованными материалами
- отчеты из диалогов
- экспериментальные заметки пользователя

## 5.2. Главные правила обработки INBOX

При каждом запуске в `documentation mode`:

1. Проверь папку `INBOX/`.
2. Найди новые или необработанные материалы.
3. Не удаляй исходники из `INBOX/` без отдельного разрешения пользователя.
4. Для каждого материала определи:

   - тему;
   - тип материала;
   - важность;
   - связанные понятия WebTutor / WebSoft HCM;
   - есть ли в нем факты, примеры, ошибки, решения, инструкции, гипотезы.

5. Сохрани копию исходника в `raw/`, разложив по дате и теме.
6. Обнови или создай материалы в `docs/`.
7. Создай или обнови атомарные заметки в `rag/`.
8. Обнови индексные файлы в `index/`.
9. В конце сформируй отчет о проделанной работе.

## 5.3. Работа с архивами

Если в `INBOX/` есть архивы:

- не удаляй архив;
- считай архив исходным материалом;
- если рядом есть распакованная версия, используй распакованную версию для анализа;
- если распакованной версии нет, можешь распаковать архив в `raw/unpacked/` или во временную рабочую область;
- не смешивай содержимое разных архивов без явной причины;
- фиксируй в `index/sources.md`, какой архив был источником.

## 5.4. Работа с PDF и изображениями

Если PDF или изображение невозможно надежно разобрать автоматически:

- не выдумывай содержание;
- создай запись в `index/open_questions.md`;
- укажи, что материал требует ручной проверки;
- если часть текста доступна, используй только доступную часть.

## 5.5. Запрет на потерю фактов

При переработке материалов:

- не удаляй важные технические детали;
- не заменяй точные имена полей общими формулировками;
- не убирай примеры кода, если они могут быть полезны;
- не превращай технический отчет в “красивую статью” с потерей фактов;
- сохраняй различие между проверенным фактом, гипотезой и TODO.

# 6. Структура docs

Папка `docs/` содержит связную документацию.

Не нужно заранее создавать идеальную структуру. Если появляется новая тема — создай новый раздел.

Возможные разделы:

```text
docs/installation/
docs/local-dev-contour/
docs/authentication/
docs/logout/
docs/request-object/
docs/response-object/
docs/tools-web/
docs/server-pages/
docs/url-routing/
docs/templates/
docs/nodes/
docs/agents/
docs/catalogs/
docs/xquery/
docs/troubleshooting/
docs/cases/
```

Если тема уже существует — обнови существующий файл, а не создавай дубль.

Каждый файл в `docs/` должен быть понятным без открытия исходного диалога.

Желательная структура документационного файла:

```markdown
---
title: ""
topic: ""
source_files: []
status: "draft"
---

# Заголовок

## Кратко

## Подробное описание

## Синтаксис / пример / сценарий

## Важные нюансы

## Проверенные факты

## TODO / вопросы для проверки

## Источники
```

# 7. Структура rag

Папка `rag/` содержит короткие атомарные RAG-ready заметки.

Один файл — один концепт, функция, ошибка, рецепт, объект, поле или проверенный кейс.

Примеры:

```text
rag/authentication/tools_web_user_init.md
rag/authentication/cookie_logout.md
rag/request/request_auth_user_id.md
rag/response/response_write.md
rag/url-routing/x-local.md
rag/server-pages/html_handlers.md
rag/agents/basic_agent_structure.md
rag/catalogs/collaborators.md
```

Каждый RAG-ready файл должен иметь frontmatter:

```markdown
---
title: ""
topic: ""
source_files: []
status: "draft"
---

# Заголовок
```

RAG-ready файл должен быть коротким, точным и пригодным для поиска.

Не смешивай в одном RAG-файле много разных тем.

---

# 8. Индексные файлы

Поддерживай файлы:

```text
index/map.md
index/sources.md
index/glossary.md
index/open_questions.md
```

## 8.1. index/map.md

Карта текущей документации.

Должна показывать:

- какие темы уже покрыты;
- где искать информацию;
- какие разделы связаны между собой;
- какие области еще пустые или слабые.

## 8.2. index/sources.md

Список обработанных источников.

Для каждого источника указывай:

- путь к исходному файлу;
- тип материала;
- тему;
- статус обработки;
- какие docs/rag файлы были созданы или обновлены на его основе.

## 8.3. index/glossary.md

Глоссарий терминов WebSoft HCM / WebTutor.

Добавляй туда технические термины, объекты, методы, поля и устойчивые понятия.

Примеры терминов:

- `Request`
- `Response`
- `Response.Write`
- `Request.AuthUserID`
- `Request.Query`
- `Request.Session`
- `Request.Redirect`
- `Env.curUserID`
- `Env.curUser`
- `tools_web.user_init`
- `tools_web.set_cookie_auth`
- `x-local://`
- `_wt`
- `collaborator`
- `agent`

## 8.4. index/open_questions.md

Список вопросов, которые нужно проверить позже.

Добавляй сюда все места, где:

- данных недостаточно;
- есть противоречие;
- нужно найти карточку объекта;
- нужно проверить точное имя поля;
- нужно проверить поведение на реальном стенде;
- нужна дополнительная документация;
- неизвестно, поддерживается ли нужная языковая конструкция внутренним серверным языком WebTutor.

# 9. Development mode

## 9.1. Назначение

В этом режиме ты решаешь конкретные задачи разработки по WebSoft HCM / WebTutor.

Примеры задач:

- написать агента;
- написать серверный HTML-хендлер;
- сделать обработчик logout;
- изменить карточку сотрудника;
- найти объект по условию;
- сформировать отчет;
- написать XQuery;
- подготовить код для портала;
- объяснить, как реализовать бизнес-логику в WT.

## 9.2. Источники для разработки

Перед реализацией задачи используй материалы из:

```text
index/map.md
index/glossary.md
index/open_questions.md
docs/
rag/
```

Если в репозитории есть релевантные материалы, опирайся на них.

Не игнорируй локальную документацию в пользу общих знаний.

## 9.3. Правило достаточности знаний

Перед тем как писать финальное решение или код, проверь, хватает ли данных.

Не придумывай недостающие детали платформы.

Если для задачи не хватает информации, остановись и задай пользователю конкретные вопросы.

Примеры недостающих данных:

- неизвестно точное имя объекта;
- неизвестно точное имя каталога;
- неизвестно точное поле;
- неизвестно значение статуса;
- неизвестен способ сохранения объекта;
- неизвестно, какой API или стандартную функцию WebTutor нужно использовать;
- неизвестно, где должен выполняться код: агент, HTML-страница, шаблон, action, кнопка, серверный метод;
- неизвестно, нужны ли права, фильтры, журналирование, транзакция или тестовый режим;
- неизвестно, можно ли менять все записи массово;
- неизвестно, нужен ли dry-run режим;
- неизвестно, как откатывать изменения;
- неизвестно, поддерживается ли нужная языковая конструкция внутренним серверным языком WebTutor.

## 9.4. Формат ответа при нехватке знаний

Если знаний не хватает, не пиши финальный код.

Ответь в таком формате:

```markdown
# Не хватает данных для безопасной реализации

## Что уже известно из документации

- ...

## Чего не хватает

- ...

## Какие материалы нужно добавить в INBOX

- ...

## Какой следующий шаг

Добавьте материалы в `INBOX/`, затем запустите `documentation mode` для обновления `docs/`, `rag/` и `index/`.

После этого можно продолжить разработческую задачу.
```

## 9.5. Формат ответа, если знаний хватает

Если знаний хватает, сначала дай краткий план реализации.

Используй формат:

```markdown
# План реализации

1. ...
2. ...
3. ...

# Важные предположения

- ...

# Код

...

# Что проверить перед запуском

- ...
```

Код должен быть максимально осторожным.

Для массовых изменений по умолчанию предлагай:

- фильтр отбора;
- логирование;
- dry-run режим;
- ограничение количества записей для теста;
- резервную копию или способ отката, если это применимо.

# 10. Связь documentation mode и development mode

Если в ходе `development mode` обнаружены пробелы в знаниях:

1. Не выдумывай ответ.
2. Сформулируй, каких данных не хватает.
3. Скажи, какие материалы нужно добавить в `INBOX/`.
4. После добавления материалов пользователь запускает `documentation mode`.
5. После обновления документации можно продолжить разработческую задачу.

Целевой цикл работы:

```text
задача по WebTutor
→ поиск в docs/rag/index
→ проверка достаточности знаний
→ если знаний хватает: решение
→ если знаний не хватает: запросить материалы
→ пользователь добавляет файлы в INBOX
→ documentation mode обновляет базу знаний
→ development mode продолжает задачу
```

# 11. Требования к стилю

Пиши технически, прямо и проверяемо.

Не используй рекламный стиль.

Не добавляй воду.

Не скрывай неопределенность.

Если факт проверен — пиши как факт.

Если это вывод из эксперимента — так и указывай.

Если это гипотеза — помечай как гипотезу.

Если нужно проверить — пиши `TODO: нужно проверить`.

Не называй серверный код WebTutor обычным JavaScript. Используй формулировку “внутренний серверный язык WebTutor” или “серверный код WebTutor”.

Сохраняй оригинальные технические имена:

- `Request`
- `Response`
- `Response.Write`
- `Request.AuthUserID`
- `Request.Query`
- `Request.Session`
- `Request.Redirect`
- `Env.curUserID`
- `Env.curUser`
- `tools_web.user_init`
- `tools_web.set_cookie_auth`
- `x-local://`
- `_wt`

# 13. Главное правило

Цель репозитория — не один раз создать идеальную документацию.

Цель — поддерживать живую базу знаний по WebSoft HCM / WebTutor, которая улучшается при каждом добавлении новых материалов и помогает решать реальные задачи разработки.
