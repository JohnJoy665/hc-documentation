# Резервное копирование и обслуживание WebSoft HCM

Надёжное функционирование WebSoft HCM требует регулярного резервного копирования и плановых работ по обслуживанию. В этом разделе собраны рекомендации из раздела «Обслуживание» базы знаний.

## Резервное копирование

Документация рекомендует придерживаться стратегии **ежедневных дифференциальных и еженедельных полноценных копий**. Основные шаги следующие【476207063959060†screenshot】:

1. **Запланируйте выполнение резервного копирования вне рабочего времени** (например, ночью), чтобы снизить нагрузку на сервер.
2. **Перед обновлением или изменениями конфигурации** сделайте полный бэкап: остановите службу WebSoft HCM, скопируйте каталог `WebSoftServer` (используйте архиватор или команду `robocopy`) и создайте дамп внешней СУБД стандартными средствами (например, `BACKUP DATABASE` для MSSQL или `pg_dump` для PostgreSQL).
3. **Храните копии на отдельном носителе** или сетевом хранилище. При использовании виртуальных машин можно делать снимки (snapshot) перед обновлениями.
4. **Проверяйте целостность резервных копий** и периодически выполняйте тестовое восстановление на отдельном стенде.

## Перезапуск службы

В случае обновления конфигураций, смены сертификатов или установки обновлений необходимо корректно перезапустить портал. В Windows это выполняется командой `net stop WebSoftHCMCorpServer` и `net start WebSoftHCMCorpServer` либо через консоль Services (см. раздел 08). В Linux перезапуск выполняется через `systemctl restart websoft-hcm.service`.

## Восстановление и обслуживание индексов

WebSoft HCM использует полнотекстовый поиск на базе Lucene. При повреждении индексов или после переноса данных необходимо их пересоздать:

1. **Отключите агент обновления индексов.** В администраторском интерфейсе найдите агент `WT_FT_UPDATE` и временно отключите его, изменив расписание на «никогда»【565386930977037†screenshot】.
2. **Удалите старые индексы.** Очистите папку с индексами (`FTIndex` или аналогичную) или переименуйте её.
3. **Пересоздайте индексы через x‑shell.** Выполните команду:

   ```bash
   rctg start
   ```

   Процесс может занять время; следите за логом `rctg.log`. После завершения появится сообщение «Indexes rebuilt»【977112133570971†screenshot】.
4. **Включите агент обратно** и убедитесь, что поиск работает корректно.

## Диагностика при падении производительности

Если портал начал работать медленно, рекомендуется собрать диагностическую информацию:

* Вызовите `/http_info` в адресной строке сервера (например, `http://portal.example.com/http_info`) и сохраните вывод. Этот отчёт содержит информацию о нагрузке, количестве потоков и времени выполнения запросов.
* Проверьте загрузку процессора и объём свободной памяти на сервере. В случае Linux можно использовать команды `top`/`htop`, в Windows — Диспетчер задач.
* Скачайте лог‑файлы `log.txt`, `errors.log`, `wtcommon.log` и передайте в техническую поддержку для анализа.
* Если подозреваете проблемы с базой данных, проверьте ожидания и блокировки на стороне СУБД (например, `sp_who2` для MSSQL).

## Дополнительные процедуры обслуживания

* **Обновляйте сервер и компоненты** до актуальных версий. Перед обновлением изучите инструкции из раздела 11 «Обновление» и обязательно сделайте резервную копию.
* **Проводите антивирусную проверку** сервера и файловых хранилищ. Убедитесь, что антивирус не блокирует исполняемые файлы WebSoft HCM.
* **Следите за свободным пространством** на дисках, особенно на разделах, где хранятся логи, временные файлы и база данных.

Своевременное обслуживание и правильная стратегия резервного копирования помогут избежать потери данных и простоев системы.
