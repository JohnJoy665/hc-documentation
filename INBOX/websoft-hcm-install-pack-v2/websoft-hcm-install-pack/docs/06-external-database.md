# Подключение к внешней базе данных

WebSoft HCM может использовать как встроенную XML‑базу, так и внешние СУБД (Microsoft SQL Server или PostgreSQL). Раздел «Внешняя БД» описывает необходимые компоненты, настройку подключения, смену кодировки и перенос данных. Операции с базой требуют опыта системного администратора и должны выполняться **только после создания полной резервной копии**【267088836628904†screenshot】.

## Компоненты для внешней БД

Начиная с релиза **1 квартал 2019** для работы с внешними базами используются компоненты **.NET Core**. Они располагаются в каталоге `WebSoftServer\dotnetcore` (`core`, `libs`, `storage`). Чтобы включить использование этих компонентов, в файле `xHttp.ini` нужно задать:

```ini
DOTNETCORE: 1
DOTNETCORE-PATH: \WebSoftServer\dotnetcore\core
DOTNETCORE-PATHS: \WebSoftServer\dotnetcore\libs
DOTNETCORE-STORAGE-PATH: \WebSoftServer\dotnetcore\storage
```

Если параметра `DOTNETCORE` нет или он равен `0`, сервер будет использовать старые .NET‑компоненты из папки `storage`【424674370675434†screenshot】.  После замены файлов перезапустите сервер.

## Изменение кодировки (collation)

При обновлении старых систем может потребоваться изменить кодировку базы данных MSSQL на **UNICODE**. Процесс состоит из нескольких шагов и выполняется на отключённом сервере:

1. **Остановите WebSoft HCM** и выполните полный бэкап СУБД.
2. Откройте файл конфигурации `spxml_unibridge_config.xml` и установите параметр `<UpgradeLocked>false</UpgradeLocked>`【977112133570971†screenshot】.
3. Запустите SQL‑скрипт `drop_all_catalog_tables.sql`, чтобы удалить таблицы журналов каталога (они будут восстановлены позже).
4. В файле `spxml_unibridge_config.xml` добавьте строку `<add key="Collation" value="UNICODE" />`【909964817883946†screenshot】.
5. Убедитесь, что в `xHttp.ini` присутствует строка `X-SHELL-START: 1` (это позволяет запускать x‑shell при старте сервера)【977112133570971†screenshot】.
6. Запустите **x‑shell** и выполните команду `rctg start`, чтобы пересоздать полнотекстовые индексы. В журнале `rctg.log` должно появиться сообщение «Indexes rebuilt».
7. После завершения индексирования верните параметр `<UpgradeLocked>true</UpgradeLocked>` и запустите сервер.

> **Важно:** Изменение кодировки — необратимая операция.  Перед выполнением скриптов убедитесь, что имеется актуальная копия базы.

## Ручное создание базы MSSQL

Если сервер БД создаётся вручную, рекомендуется использовать поставляемый скрипт `create_db.sql`. В нём можно изменить имя базы и пути к файлам данных и журналов. Статья «Мануал по созданию базы MSSQL вручную» содержит лишь общее описание и предполагает, что операции выполняет администратор SQL Server【405210900840631†screenshot】.

## Перенос данных в MSSQL

Для перехода со встроенной XML‑базы или другой СУБД на MSSQL используйте x‑shell. Общая последовательность выглядит так:

1. Запустите x‑shell: в режиме Docker это делается через `docker exec -it hcm x-shell`, в Windows — через `xhttp_64.exe /s`.
2. Укажите тип базы данных: `db set type mssql`【652070062790714†screenshot】.
3. Создайте новую базу: `db create`【410804710294610†screenshot】.
4. Инициализируйте схему: `db init`.
5. Перенесите данные: `db migrate from xml` (если исходная база — встроенная XML) или `db migrate from pgsql` / `db migrate from oracle` при миграции с других СУБД【869171920895337†screenshot】.
6. Измените параметры в `spxml_unibridge_config.xml`: установите `<UpgradeLocked>false</UpgradeLocked>` и `<InPlaceUpgrade>0</InPlaceUpgrade>`.
7. Пересоздайте полнотекстовые индексы: `rctg start` и дождитесь окончания процесса【159708575601300†screenshot】.
8. Верните `<UpgradeLocked>true</UpgradeLocked>` и перезапустите сервер.

В завершение рекомендуется выполнить команду T‑SQL:

```sql
ALTER DATABASE [WTDBNAME] SET AUTO_UPDATE_STATISTICS_ASYNC ON;
```

Это позволит СУБД самостоятельно актуализировать статистику после миграции.

## Always On (кластеры MSSQL)

Для работы с SQL Server Always On используйте адрес слушателя группы доступности (availability group listener) в параметре `<Server>` конфигурации `spxml_unibridge_config.xml`. Дополнительно необходимо задать параметры:

* `MultiSubnetFailover=true` — оптимизирует переключение в кластере【997588260766744†screenshot】.
* `AlwaysOnApplicationIntentStrategy=true` — разрешает чтение с реплик и подключение к слушателю【181089438120815†screenshot】.
* `AlwaysOnReadOnlyInstances=<имена_реплик>` — список реплик, которые будут использоваться для чтения (например, для отчётов)【484202670071803†screenshot】.
* `ReadOnlyConnectionSplitStrategy=<strategy>` — определяет стратегию распределения запросов на чтение (round‑robin, random и др.).

После внесения изменений перезапустите сервер, чтобы подключения использовали настройки Always On.

## PostgreSQL

Поддерживается также PostgreSQL. В документации отмечено, что размер содержимого в поле **CDATA** веб‑карт может быть ограничен 10 МБ, поэтому изображения для карточек объектов следует сжимать до 7,5 МБ【628110361462685†screenshot】. Процесс миграции данных схож с MSSQL и выполняется через x‑shell:

1. Запустите x‑shell (в контейнере или локально) и укажите тип БД: `db set type pgsql`.
2. Создайте базу: `db create`.
3. Инициализируйте: `db init`.
4. Перенесите данные: `db migrate from mssql` или `db migrate from xml` (в зависимости от источника).
5. Запустите индексацию: `rctg start`.

На странице также упомянуты варианты миграции из MSSQL в PostgreSQL и настройка подключения по сертификату; конкретные шаги аналогичны приведённым выше и подразумевают наличие прав администратора базы【149110056019404†screenshot】.

## Общие рекомендации

* Все операции по созданию и миграции баз выполняйте на **тестовом стенде** и только после бэкапа.
* Изменение конфигурационных файлов следует сопровождать перезапуском служб WebSoft HCM.
* После миграции проверьте индексы, каталоги и права доступа. При необходимости запустите утилиты обслуживания (см. 10‑backup‑and‑maintenance.md).
